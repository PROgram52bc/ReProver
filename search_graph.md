# Qualitative Analysis with Search Path Graphs

This guide explains how to identify interesting theorems and visualize the prover's search behavior using search path graphs. This is essential for understanding exactly how the repair model helps (or hinders) the proof process.

## 1. Identifying Target Theorems
Before visualizing, you need to identify which theorems are worth analyzing—specifically those where the baseline and repair runs produced different results.

### Using `compare_outcomes.py`
Use this script to compare two global log files (e.g., your baseline run and your gAPRIL run).

```bash
python compare_outcomes.py baseline.log repair.log --output comparison.csv
```

### Understanding the Output
The script produces a **COMPARISON SUMMARY** in the terminal:
- **Gains**: Theorems that failed in Log 1 but were solved in Log 2. These represent the "success stories" of your repair model.
- **Regressions**: Theorems that were solved in Log 1 but failed in Log 2. These often indicate cases where the repair model's overhead caused a timeout or led the search into a dead end.

The `comparison.csv` file provides a theorem-by-theorem breakdown, allowing you to pick a specific theorem name (e.g., `mathd_algebra_31`) for deep-dive visualization.

## 2. Generating Detailed Logs for a Theorem
Once you have identified a theorem of interest, you need to produce a clean log file containing only the search steps for that specific theorem.

### Option A: Automatic Generation (Recommended)
Run `prover/evaluate.py` using the **exact same configuration** as your original run, but add the `--full-name` argument:

```bash
python prover/evaluate.py \
    --data-path data/minif2f \
    --dataset minif2f \
    --gen_ckpt_path kaiyuy/leandojo-lean4-tacgen-byt5-small \
    --repair-ckpt-path uw-math-ai/gAPRIL-wo-exp \
    --repair-count 2 \
    --full-name "mathd_algebra_31"
```

### Option B: Manual Extraction
If you don't want to re-run the search, you can manually extract the relevant lines from an existing global log file. Search for the line `Proving Theorem(...)` for your target theorem and copy everything until the next `Proving Theorem` line or the `SearchResult` summary.

**Example using `grep` to find the exact range:**
```bash
# 1. Find the start of your theorem AND the start of the next one
grep -n "Proving Theorem" global.log | grep -A 1 "mathd_algebra_31"

# This will output something like:
# 500:Proving Theorem(..., full_name='mathd_algebra_31')
# 1200:Proving Theorem(..., full_name='mathd_algebra_32')

# 2. Extract the range (using the numbers from above)
sed -n '500,1199p' global.log > single_thm.log
```

## 3. Visualizing the Search Tree
Now, convert that log file into a graphical format that humans can read.

### Step 1: Run the Plotting Script
Use `plot_search_tree.py` to parse the log and generate Mermaid.js code:

```bash
python plot_search_tree.py logs/trace_20260401_123456.log --output tree.mmd
```

### Step 2: Render the Graph
1. Open the generated `tree.mmd` file and copy its contents.
2. Go to the [Mermaid Live Editor](https://mermaid.live/) (or [Mermaid.ai](https://mermaid.ai/app/dashboard)).
3. Paste the code into the editor.

### Interpreting the Graph
- **Standard Nodes (Blue)**: These represent proof states (Lean goals). The text shows a truncated version of the goal.
- **Green Nodes (`FINISH_X`)**: These represent successful completion of the proof.
- **Red Nodes (`ERROR_X`)**: These represent Lean compiler errors.
- **Edges**: The arrows represent tactics.
    - **Solid Lines**: Tactics from the primary generator.
    - **Labeled `(REPAIR)`**: Tactics generated by the repair model.
- **Repair Chains**: If you used `--repair-count > 1`, you will see chains like:
  `Node` $\xrightarrow{failed\_tac}$ `Error 1` $\xrightarrow{repair\_1}$ `Error 2` $\xrightarrow{repair\_2}$ `Success`.
  This allows you to see exactly how many attempts it took for the model to self-correct.
